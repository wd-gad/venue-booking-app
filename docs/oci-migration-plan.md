# OCI Migration Plan

## Goal

Supabase Cloud and local Supabase dependency will be removed.

Target outcomes:

- Database runs on Oracle Cloud Infrastructure (OCI).
- Auth is owned by the application, not Supabase.
- File storage moves to OCI Object Storage.
- Environments are managed from one control plane.
- Public access is protected by Cloudflare Tunnel and Cloudflare Access.

## Recommendation

Use this target stack:

- App runtime: Next.js on Vercel or OCI Compute
- Database: self-hosted PostgreSQL on OCI Compute
- Auth: Better Auth
- File storage: OCI Object Storage
- Email: OCI Email Delivery (SMTP) or external SMTP
- Edge access: Cloudflare Tunnel + Cloudflare Access when app runs on OCI

## Why This Stack

### Self-hosted PostgreSQL on OCI Compute

The current application is PostgreSQL-shaped. Replacing Supabase with Oracle Autonomous Database would force a larger rewrite around SQL behavior, JSON handling, and migration tooling.

For this project, cost matters more than managed convenience. `OCI Compute + PostgreSQL` is the better baseline because:

- multiple environments are expected
- OCI Always Free Compute can host PostgreSQL directly
- the schema can later be moved to managed PostgreSQL if needed

References:

- https://docs.oracle.com/en-us/iaas/Content/postgresql/
- https://www.oracle.com/cloud/postgresql/

### Better Auth

The current product needs:

- email + password
- magic link / OTP
- Google SSO
- session management

Better Auth supports these directly and works with PostgreSQL.

References:

- https://www.better-auth.com/docs/basic-usage
- https://www.better-auth.com/docs/authentication/email-password
- https://www.better-auth.com/docs/plugins/magic-link
- https://www.better-auth.com/docs/adapters/drizzle
- https://www.better-auth.com/docs/adapters/postgresql

### OCI Object Storage

OCI Object Storage exposes an S3-compatible API, which keeps application-side file upload logic straightforward.

References:

- https://docs.oracle.com/iaas/Content/Object/Tasks/s3compatibleapi.htm
- https://docs.oracle.com/en-us/iaas/Content/Object/s3-virtual-style.htm

### Why no management layer first

The first goal is to get a stable, cheap baseline on OCI.

Start with:

- Docker Engine
- Docker Compose
- one or two OCI Compute instances

Only revisit Coolify or Dokploy after the application is stable without them. A management UI should not be the first moving part.

References:

- https://coolify.io/docs
- https://coolify.io/docs/knowledge-base/server/introduction
- https://coolify.io/docs/get-started/installation
- https://coolify.io/docs/knowledge-base/docker/compose
- https://docs.dokploy.com/docs/core/docker-compose
- https://docs.dokploy.com/docs/core/remote-servers
- https://docs.dokploy.com/docs/core/remote-servers/validate

## Implementation Phases

### Phase 1: Database layer decoupling

Replace direct Supabase table access with an application-owned data layer.

Tasks:

- introduce Drizzle ORM
- define schema for:
  - venues
  - venue_notes
  - venue_documents
  - auth tables generated by Better Auth
- move all venue CRUD to repository functions
- remove direct `.from("venues")` calls from UI components

Recommended packages:

- `drizzle-orm`
- `pg`
- `drizzle-kit`

Reference:

- https://orm.drizzle.team/docs/get-started/postgresql-new

### Phase 2: Auth replacement

Replace Supabase Auth with Better Auth.

Tasks:

- add Better Auth server config
- add Google provider
- add email/password
- add magic link or email OTP
- replace `supabase.auth.*` calls in UI
- replace auth callback route
- replace middleware/session refresh logic

### Phase 3: Storage replacement

Replace Supabase Storage with OCI Object Storage.

Tasks:

- create `venue-documents` bucket
- implement upload helper using S3-compatible API
- implement signed/public URL strategy
- update permit upload flow

### Phase 4: Infrastructure rollout

Provision OCI and put deployment in place.

Tasks:

- create OCI Compute for self-hosted PostgreSQL
- create OCI Compute or Vercel target for app runtime
- create Object Storage bucket
- create OCI Email Delivery SMTP credentials
- add Docker Compose based deployment flow

### Phase 5: Traffic cutover

Tasks:

- attach app domain
- route through Cloudflare Tunnel
- protect app with Cloudflare Access
- migrate existing venue data
- disable Supabase paths

## Data Model Notes

Current `conversation_notes` is embedded JSON on the venue record.

Recommendation:

- move notes into a first-class `venue_notes` table
- move permit file metadata into a first-class `venue_documents` table

Reason:

- edit/delete/like history is growing
- row-level update conflicts become harder with a JSON blob
- auditability and indexing are better with dedicated tables

## Migration Risks

1. Auth rewrite risk
   - highest risk area
   - all sign-in flows will change at once

2. Storage URL semantics
   - Supabase public URL behavior differs from S3-compatible signed URLs

3. Embedded JSON note structure
   - should be normalized before long-term growth

4. Existing Cloudflare Tunnel layout
   - current `supabase.*` hostname becomes obsolete
   - future public surface should be only the app hostname

## Recommended Order Of Work

1. Add Drizzle and move venue CRUD behind repositories.

3. Swap UI auth calls from Supabase to Better Auth.
4. Replace file upload with OCI Object Storage.
5. Provision OCI and deploy through Docker Compose.
6. Remove `src/lib/supabase/*` and Supabase environment variables.
